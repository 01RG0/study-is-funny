I ran into the same error. The problem is the model ID that Dify uses when it verifies the API token. You can clear the error by changing the model ID in google.py inside the API service.
[https://github.com/langgenius/dify/blob/0.15.3/api/core/model_runtime/model_providers/google/google.py](https://github.com/langgenius/dify/blob/0.15.3/api/core/model_runtime/model_providers/google/google.py)

```
import logging from core.model_runtime.entities.model_entities import ModelType from core.model_runtime.errors.validate import CredentialsValidateFailedError from core.model_runtime.model_providers.__base.model_provider import ModelProvider logger = logging.getLogger(__name__) class GoogleProvider(ModelProvider): def validate_provider_credentials(self, credentials: dict) -> None: """ Validate provider credentials. If validation fails, raise an exception. :param credentials: provider credentials defined in `provider_credential_schema`. """ try: model_instance = self.get_model_instance(ModelType.LLM) # Use `gemini-1.5-pro` for validation instead of the default. model_instance.validate_credentials(model="gemini-1.5-pro", credentials=credentials) except CredentialsValidateFailedError as ex: raise ex except Exception as ex: logger.exception( f"{self.get_provider_schema().provider} credentials validation failed" ) raise ex
```

I also wrote a (Japanese-language) blog post about the issue—feel free to use it as a reference:
[https://zenn.dev/upgradetech/articles/dify-old-version-use](https://zenn.dev/upgradetech/articles/dify-old-version-use)

For those running Dify 0.15.3 in Docker and encountering the 404 error when configuring the Google Gemini API (models/gemini-pro is not found), the solution provided by [@cH6noota](https://github.com/cH6noota) to modify google.py works great, but it won’t work directly in a Dockerized environment. Since Docker containers are ephemeral, you can’t just edit the source code inside the container (e.g., /app/api/core/model_runtime/model_providers/google/google.py) as changes will be lost on restart. Instead, you need to build a custom Docker image based on langgenius/dify-api:0.15.3 with the modified google.py file.
Here’s how to do it:
Clone the Dify repository to get the source code for version 0.15.3: bash

```
git clone --branch 0.15.3 https://github.com/langgenius/dify.git cd dify
```


```
git clone --branch 0.15.3 https://github.com/langgenius/dify.git cd dify
```

Modify the google.py file located at api/core/model_runtime/model_providers/google/google.py. Update the validate_provider_credentials method to use gemini-1.5-pro (or another supported model like gemini-1.5-flash) instead of gemini-pro. The relevant section should look like: python

```
try: model_instance = self.get_model_instance(ModelType.LLM) # Use `gemini-2.5-pro` for validation or use other valid model. model_instance.validate_credentials(model="gemini-2.5-pro", credentials=credentials) except CredentialsValidateFailedError as ex: raise ex
```


```
try: model_instance = self.get_model_instance(ModelType.LLM) # Use `gemini-2.5-pro` for validation or use other valid model. model_instance.validate_credentials(model="gemini-2.5-pro", credentials=credentials) except CredentialsValidateFailedError as ex: raise ex
```

Create a Dockerfile in the dify directory to build a custom image. Below is the Dockerfile content:
Build the custom image:
bash

```
docker build -t custom-dify-api:0.15.3 .
```


```
docker build -t custom-dify-api:0.15.3 .
```

Update your docker-compose.yaml (located in dify/docker/docker-compose.yaml) to use the custom image for the api and worker services: yaml

```
services: api: image: custom-dify-api:0.15.3 # ... other configurations ... worker: image: custom-dify-api:0.15.3 # ... other configurations ...
```


```
services: api: image: custom-dify-api:0.15.3 # ... other configurations ... worker: image: custom-dify-api:0.15.3 # ... other configurations ...
```

Restart the containers: bash

```
cd docker docker compose down docker compose up -d
```


```
cd docker docker compose down docker compose up -d
```

Verify the fix in the Dify dashboard under Settings > Model Provider > Google Gemini. Enter your API key and save. The 404 error should be resolved. Notes:
Ensure your Gemini API key has access to gemini-1.5-pro or gemini-1.5-flash. Check available models with: bash

```
curl "https://generativelanguage.googleapis.com/v1beta/models?key=YOUR_API_KEY"
```


```
curl "https://generativelanguage.googleapis.com/v1beta/models?key=YOUR_API_KEY"
```

This approach ensures changes persist across container restarts, unlike mounting a modified file via volumes. If you’re in a region with network restrictions (e.g., China), configure proxy settings in docker-compose.yaml as needed. Upgrading to a newer Dify version might fix this issue without modification, but test carefully as some versions may introduce other challenges. This solution worked for me, and I hope it helps others running Dify in Docker! Thanks to [@cH6noota](https://github.com/cH6noota) for the original fix.

Solution Found: 404 v1beta error is caused by an incorrect model name alias. After extensive debugging, I've found the solution to the persistent 404 Not Found for API version v1beta error.
The Problem: My script was failing when calling generate_content with a 404 error pointing to a deprecated model (...-002) and the v1beta API, even though my code was using what I thought was the correct model name.
The Solution: The issue was a simple but crucial error in the model name. The correct alias for the latest Flash model is not gemini-1.5-flash-latest.
The fix is to change the model name in the code:
OLD (Incorrect): MODEL_NAME = 'gemini-1.5-flash-latest'
NEW (Correct): MODEL_NAME = 'gemini-flash-latest'
This single change completely resolved the error on a stable Python environment (3.11.9).
Additional Context for Advanced Users (e.g., Arch Linux on Python 3.13+):
For users on bleeding-edge systems, please note that this model name issue was compounded by a separate library incompatibility with the new Python 3.13.7. On that version, a faulty version of the google-generativeai library was being installed, which made debugging this much harder. The ultimate solution for that environment was to use pyenv to manage a stable Python version (like 3.11 or 3.12) OR apply a specific dependency fix (pinning grpcio versions).
However, for most users on stable Python versions, simply correcting the model name to gemini-flash-latest should be the definitive solution.
Closing this issue as resolved.

The "404 models/gemini-pro is not found" error occurs because Google's Gemini API uses different model naming conventions and requires proper API configuration. Here's how to fix it:
1. Update to Correct Gemini Model Names:
Google has deprecated gemini-pro. Use these current models instead:

```
gemini-pro
```


```
gemini-1.5-pro # Latest high-performance model gemini-1.5-flash # Fast and efficient model gemini-2.0-flash # Latest available model (if enabled) gemini-1.0-pro # Legacy but still available
```


```
gemini-1.5-pro # Latest high-performance model gemini-1.5-flash # Fast and efficient model gemini-2.0-flash # Latest available model (if enabled) gemini-1.0-pro # Legacy but still available
```

2. Verify Your API Setup in Dify:
- Go to: Settings > Model Providers > Google Gemini
- Select the correct model from the dropdown
- Ensure your API key has access to the model
- Check that the API key is from a Google Cloud project with Generative AI API enabled
3. Update google-generativeai Library:
If you're using Dify with custom models, update the library:

```
pip install --upgrade google-generativeai>=0.4.0
```

4. Check API Configuration:
Verify these settings are correct:

```
API Key: Your Google Generative AI API key (from ai.google.dev or Google Cloud) Model: gemini-1.5-pro or gemini-1.5-flash Version: v1 (NOT v1beta - v1beta is deprecated for production use)
```


```
API Key: Your Google Generative AI API key (from ai.google.dev or Google Cloud) Model: gemini-1.5-pro or gemini-1.5-flash Version: v1 (NOT v1beta - v1beta is deprecated for production use)
```

5. Common Mistakes to Avoid:
- ✗ Using gemini-pro (deprecated)
- ✗ Using v1beta API version (use v1 instead)
- ✗ Using wrong API key (ensure it's Generative AI API key, not Cloud key)
- ✗ Model not enabled in your Google Cloud project

```
gemini-pro
```


```
v1beta
```


```
v1
```

6. For Dify Docker Setup:
If self-hosting, ensure your environment variables are set:

```
# In docker/.env GOOGLE_API_KEY=your_api_key_here GOOGLE_API_VERSION=v1
```

Restart containers:

```
docker-compose down docker-compose up -d
```

7. Test Connection:
Create a simple test workflow:
1. Add an LLM node
2. Select Google Gemini
3. Choose gemini-1.5-flash
4. Enter a simple prompt: "Say hello"
5. Run and verify it works

```
gemini-1.5-flash
```

Why You're Seeing This Error:
Google sunsetted gemini-pro as it was a preview model. The current production models are gemini-1.0-pro, gemini-1.5-pro, and gemini-1.5-flash. The error also suggests you might be using an old API version (v1beta) which doesn't support newer models.

```
gemini-pro
```


```
gemini-1.0-pro
```


```
gemini-1.5-pro
```


```
gemini-1.5-flash
```


```
v1beta
```

If Still Having Issues:
1. Verify API key: Test at [https://ai.google.dev/](https://ai.google.dev/)
2. Check project quota: Google Cloud Console > Quotas
3. Enable API: Google Cloud > Generative AI API (enable if not)
4. Check billing: Ensure your project has active billing
[https://ai.google.dev/](https://ai.google.dev/)
Switch to gemini-1.5-flash for best results - it's faster and more reliable than the old gemini-pro.

```
gemini-1.5-flash
```


```
gemini-pro
```
